Create Custom Reports

Security

As of version 2018.5.0, security was implemented in allowing the Administrator to create groups, assign users to those groups, and grant permissions to existing reports as well as custom reports. In addition, a new field was added that corresponds with the named permission required to see and run the report.

About Custom Reports

Administrators with a strong SQL background can use stored procedures to create custom reports for inclusion in the Reports component.

The following figure shows a custom report that has been included in Reports.

Creating custom reports can be as simple as creating a stored procedure that returns the data for a report (see Create a Stored Procedure), or you can define charts and graphs and fine-tune the layout of the data (see Create a Report Definition).

Note the following when creating a custom report for :

  • By default, report data is displayed as a table with the columns in the same order as they appear in the resulting dataset(s).

  • Multiple datasets can be returned from the same stored procedure and will be displayed when the report is run.

  • The reporting system will automatically aggregate any fields containing numbers. Do not include IDs, codes, or other numeric fields that you do not want aggregated. (This behavior can be overridden by customizing the data display as described in the Reporting.ParameterListDefinition Table topic.)

  • If you are planning on including charts in your report, be aware that extremely large datasets may not display well in charts.
